Artifact Summary

CS-499-worker is a .NET 8.0 worker service that uses AMQPNetLite to subscribe to multiple AMQ topics from a message broker. It processes the messages, extracts a few entities, and adds them to tables in the "SMAT" database (CS-499-database). It also checks the message barcode in the database and inserts the entities with "New" status if not found, or "Updated" if found.

Data Structures and Algorithms Enhancement

Original Artifact Enhanced Artifact Video Demo
Justification

The legacy system that some of the business logic and database code was pulled from has a history of concurrency issues and dropped messages. For that reason, I wanted to pay extra close attention to how data is being accessed, stored, and updated or deleted. The barcode lookup and lock, in particular, caused issues when multiple messages were received at the same time and processed at the same time on multiple threads or servers. The need to quickly search for the barcode and handle it concurrently made it a good candiate for the Data Structures and Algorithms enhancement.

  • C# / .NET 8
  • Algorithm Design
  • Concurrent Processing
Enhancements Made

In this enhancement, I implemented an in-memory cache for storage of frequent barcode lookups, and created two versions of a barcode lookup algorithm and/or data structure. The first was a binary search algorithm, and the second made use of the ConcurrentDictionary data structure. I then created unit tests to compare the performance of each lookup.

  • Enhancement 1: Add in-memory cache.
  • Enhancement 2: Add barcode lookup service, binary search version, and ConcurrentDictionary version.
  • Enhancement 3: Add unit test to compare lookup time.
  • Enhancement 4: Continue adding additional business logic and logging improvements.

Database Enhancement

Original Database Initial Naming Rewrite AlloyDB Rewrite Original Worker Enhanced Worker Video Demo
Justification

The service is part of a much larger ecosystem of microservices at my company. It was created a few months ago both for business requirements and as a POC for adoption of new pipeline systems, and I worked with various teams to implement and deploy it via enterprise pipelines using Azure DevOps, Openshift, and ArgoCD. However, the enterprise is now switching to GCP, Cloud Run, and AlloyDB, and because we are planning to use AI solutions to read data, the legacy tables and columns must be renamed to be more readable (this will also help the humans). For that reason, the application must be tweaked for for those platforms, and the database must be rewritten from SQL Server to AlloyDB/PostgreSQL. That made it a perfect candiate for the Database enhancement.

  • SQL Server / AlloyDB / GCP / Terraform
  • Migrate and Modernize Data

I worked with our database design teams to understand the meaning of the table and column abbreviations and adopt/define new standards for naming, and rewrote the tables, columns, and stored procedures for AlloyDB/PostgreSQL. I began setting up the clusters, instances, databases, networking, service accounts, etc, as well. Infrastructure work has required outreach/dependency on external teams, and as a result is still in progress.
